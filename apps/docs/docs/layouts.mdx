---
title: Layout System
description: How to organize schema fields into forms and tables in Nubase.
---

Nubase schemas have two optional layout attachments:

- **Form layout** — how fields are grouped, ordered, and sized when rendered as a form. Attach via `withFormLayout`.
- **Table layout** — which fields appear as columns (with widths, pinning, and inline-edit flags) when rendered as a table or grid. Attach via `withTableLayout`.

Both are optional. Without them, the form/table renderers fall back to a default: every field in a single full-width group (form) or no columns at all (table).

## Form layouts

```typescript
import { nu } from "@nubase/core";

const ticketSchema = nu
  .object({
    title: nu.string().withComputedMeta({ label: "Title" }),
    description: nu.string().withComputedMeta({ label: "Description" }),
    priority: nu.number().withComputedMeta({ label: "Priority" }),
    status: nu.string().withComputedMeta({ label: "Status" }),
    assignee: nu.string().withComputedMeta({ label: "Assignee" }),
    dueDate: nu.string().withComputedMeta({ label: "Due Date" }),
  })
  .withFormLayout({
    groups: [
      {
        label: "Basic Information",
        fields: [
          { name: "title", fieldWidth: 12 },
          { name: "description", fieldWidth: 12 },
        ],
      },
      {
        label: "Assignment",
        fields: [
          { name: "priority", fieldWidth: 6 },
          { name: "status", fieldWidth: 6 },
          { name: "assignee", fieldWidth: 6 },
          { name: "dueDate", fieldWidth: 6 },
        ],
      },
    ],
  });
```

### Groups

Each group has:

```typescript
{
  label?: string;          // optional heading shown above the group
  description?: string;    // optional subtitle
  fields: FormLayoutField[]; // the fields in this group
  className?: string;      // optional className for the group container
  collapsible?: boolean;   // whether the group can collapse
  defaultCollapsed?: boolean;
}
```

### Fields

Each field entry is:

```typescript
{
  name: keyof TShape;    // type-checked against the schema shape
  fieldWidth?: number;   // 1–12 grid units (12 = full width)
  className?: string;
  hidden?: boolean;
}
```

`fieldWidth` is a 12-column grid — `6` is half-width, `4` is a third, `12` is full width. Fields default to full width when omitted.

Field order follows the order in the layout. Fields not listed in any group are not rendered.

## Table layouts

```typescript
const ticketListSchema = ticketSchema.withTableLayout({
  fields: [
    { name: "id", label: "ID", columnWidthPx: 80, pinned: true },
    { name: "title", label: "Title", columnWidthPx: 300, editable: true },
    { name: "status", label: "Status", columnWidthPx: 150 },
  ],
  metadata: {
    patchable: true, // enable inline editing via onPatch on the search view
  },
});
```

### Field options

```typescript
{
  name: keyof TShape;          // type-checked against the schema shape
  label?: string;              // column header (defaults to field name)
  columnWidthPx?: number;      // explicit pixel width
  pinned?: boolean;            // freeze to the left of the grid
  editable?: boolean | "auto-commit"; // inline editing mode
  hidden?: boolean;
  className?: string;
}
```

### Metadata

```typescript
{
  patchable?: boolean;         // enable inline patching across the table
  linkFields?: (keyof TShape)[]; // fields whose values are navigation links
}
```

## Best practices

- Keep one `withFormLayout` per schema — it describes the canonical structure.
- Use `fieldWidth` to collapse natural pairs (e.g. two half-width fields on one row) instead of creating a group per row.
- Put virtual fields from `nu.relation(...)` in their own group — they render as a section (table with its own heading) and are usually full width.
