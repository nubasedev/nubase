The typed-sql pipeline is exposed as a single CLI subcommand:

```
nubase data-layer generate [options]
```

## Configuration

The command reuses the existing [nubase.config.ts](/docs/configuration) environment system — it reads `DATABASE_URL` (or whatever you've wired into your environment's `url` field) the same way `nubase db pull` and `nubase db push` do.

Add an optional `dataLayer` section to point at your `data-layer/` root:

```ts title="nubase/nubase.config.ts"
import { defineConfig } from "@nubase/cli";

export default defineConfig({
  environments: {
    local: { url: process.env.DATABASE_URL! },
    prod: { url: process.env.QUESTLOG_PROD_DATABASE_URL! },
  },
  defaultEnvironment: "local",
  schemas: ["public"],
  dataLayer: {
    dir: "src/backend/data-layer", // default; set to override
  },
});
```

The `dataLayer.dir` path is resolved relative to the **app root** (the parent of the `nubase/` config directory), matching how the runtime resolves other paths. If you omit the `dataLayer` section entirely, the default is `src/backend/data-layer`.

## Dev mode — writing files

```bash
npx nubase data-layer generate
```

What happens:

1. Load `nubase.config.ts` and resolve the environment.
2. Parse the `DATABASE_URL` into host/port/user/password/db/ssl.
3. Walk `data-layer/` recursively for `.sql` files (skipping hidden files, `node_modules/`, `dist/`, and basenames that aren't valid JS identifiers).
4. Open **one** long-lived Postgres wire connection — not one per query.
5. For each file: preprocess `:named` params, send `Parse` + `Describe`, resolve OIDs via `pg_type`/`pg_attribute`/`pg_description`, render the TypeScript output.
6. Write each generated `.ts` file **only if its content differs** from what's already on disk — this keeps mtimes stable so Vite/tsc don't re-process unchanged files.
7. Delete orphaned `.ts` files whose `.sql` sibling no longer exists (only files with the `Generated by @nubase/pg` header are touched — hand-written `.ts` files are preserved).
8. Print a summary and exit.

**Error recovery:** if a query fails to parse, the CLI prints its diagnostic, skips to the next query, and exits with code 1 at the end. You see every error at once, not just the first one. Failed queries do not overwrite their previous `.ts` file — so `tsc` doesn't break on a partially-written file.

### Successful output

```
→ Generating data-layer .ts files in /Users/.../src/backend/data-layer
✓ 18 generated, 2 unchanged, 1 orphans removed (342ms)
```

### Failing output

```
→ Generating data-layer .ts files in /Users/.../src/backend/data-layer

✖ tickets/listOpenTickets.sql
  42703: column "titel" does not exist

     4 │   t.id,
     5 │   t.titel,
       │     ^
     6 │   t.status
       │
       │ Hint: Perhaps you meant to reference the column "tickets.title".
       │
       │ ↪ Postgres error code 42703 (undefined_column)

✗ 1 queries failed, 17 ok (329ms)
```

The diagnostic includes:

- The `.sql` file path, relative to the project root (clickable in most terminals).
- The SQLSTATE code and message from Postgres.
- Two lines of context above and one line below the error, with a caret pointing at the error column.
- Postgres' own hint, if present.
- The SQLSTATE code's human-readable name (from an internal ~40-entry table of common errors; unknown codes fall back to just the number).

The position is translated from the rewritten SQL back to the original source via a source map built by the `:name` → `$N` preprocessor, so the caret lines up with the original file — not with the rewritten query Postgres saw.

## CI mode — `--check`

```bash
npx nubase data-layer generate --check
```

`--check` mode runs the same pipeline but:

- **Doesn't write any files.** The generated content is produced in memory.
- **Compares source hashes** against the `// Hash: sha256:...` header in each existing generated `.ts` file. Any mismatch means the `.sql` was edited but the `.ts` wasn't regenerated.
- **Surfaces orphans as failures** instead of deleting them. A `.sql` file that was deleted without running the generator afterwards will be flagged.
- **Runs Parse/Describe for every query**, so stale queries that reference missing columns or dropped tables are caught too.
- **Exits with code 1** on any failure: parse errors, stale hashes, or orphans.

This is the gate you want in CI. Add a step like:

```yaml title=".github/workflows/ci.yml"
- name: Start postgres
  run: docker compose up -d postgres

- name: Apply migrations
  run: npx nubase db push --yes

- name: Check typed-sql generated files
  run: npx nubase data-layer generate --check
```

If any developer edits a `.sql` without running the generator, or if the live schema drifts such that a committed query no longer parses, CI fails on the PR.

### Why does `--check` need a live database?

The strict alternative would be a hash-only comparison — "do the committed `.ts` files match the committed `.sql` files?" That catches "dev forgot to regenerate" but not "the regenerated file would no longer parse against the current schema."

Requiring the live DB makes `--check` the strongest possible guarantee: the committed `.ts` matches the committed `.sql`, *and* the queries still parse against the current schema. This is usually what you want — if a migration broke a query, CI should catch it the same way it catches a broken type.

### Hash scheme

The hash in the `.ts` header is `sha256(sourceBytes)` truncated to 16 hex characters. It covers the **source `.sql` only**, not the generated output. That's deliberate:

- Upgrading `@nubase/pg` shouldn't invalidate every file in every project.
- The check answers the right question: "does this `.ts` correspond to this `.sql`?", not "was this `.ts` produced by the exact current generator version?"

If you later want a version gate too, we can add a `Generator:` header line. It's deferred for now.

## Options reference

```
Usage:
  nubase data-layer generate [options]

Options:
  --env <name>      Target environment from nubase.config.ts (default: "local")
  --check           Don't write anything; exit 1 if any file is stale or fails to parse
  --help            Show help
```

## Mental model

Run `nubase data-layer generate` after you:

- Edit any `.sql` file (even a whitespace change will change its hash and need regeneration).
- Edit `@json` annotations.
- Delete a `.sql` file (to clean up the orphaned `.ts`).
- Apply migrations that change the shape of any table a query references.
- Add a new `.sql` file.

Run `nubase data-layer generate --check` in CI after the database is up and migrations are applied. Nothing else needs to be added to your CI pipeline.
