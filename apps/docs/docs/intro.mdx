## What is Nubase?

Nubase is a highly-opinionated, batteries-included, meta-framework for building internal tools with TypeScript. 
Nubase is not designed to be a general-purpose framework. Instead, it focuses on applications centered around CRUD operations (Create, Read, Update, Delete),
business logic, workflows, dashboards and configuration pages.

Nubase is a good fit for domain-centric applications, tools whose sole purpose is to manipulate data within a well-defined domain. Salesforce, for instance, exists entirely to manage customer relationship data.

Think of Nubase as an open-source alternative to platforms like Retool and Airtable, except that it's open-source and
you have full control over the database and the business logic. You don't have full control over the frontend, and the wiring
between the frontend and your business logic. And that's the intentional trade-off we will discuss in the next section.


## Why Nubase?

Every modern internal tool has a common set of requirements:
- Authentication and authorization.
- Dashboards.
- Searching, filtering, viewing and editing data.
- Business logic that happens when data is created, updated or deleted.
- Configuration pages.
- Theming.
- A frontend infrastructure that supports menus, navigation, forms, tables, and other common UI components, and 
non-functional requirements like responsiveness, accessibility, and internationalization.
- An AI infrastructure that supports actions to be executed by AI agents, and a UI for users to interact with those agents.

Teams often reinvent these core components for each new project. This leads to wasted development cycles, inconsistent user experiences, and a gradual drift away from modern UX and accessibility standards.

This highlights a fundamental trade-off in software engineering: flexibility versus constraint. A highly flexible system allows for tailored solutions but sacrifices development speed and consistency. A rigid, constrained system is fast and consistent but can't adapt to unique requirements, forcing compromises in the user experience.

Nubase is built on a strong conviction: With the right architecture, by deliberately choosing low-flexibility we can achieve exceptional gains in development speed, consistency, and out-of-the-box quality. Our architecture is meticulously designed to optimize for this approach.

We are explicit about this trade-off. Nubase is designed to excel at the 85% of common business application requirements. We acknowledge that this focus means the remaining 15% of highly specific or unique features will have to be adapted and there will be compromises. We believe that for the vast majority of business applications, the benefits of super fast development, guaranteed consistency, and built-in quality far outweigh the limitations of this focused approach.

## Which constraints does Nubase impose?

A typical Nubase app uses this stack:
- Node/TypeScript for the full-stack app.
- Postgres is the main data-store, accessed through the Kysely query builder. Nubase API endpoints are TypeScript
functions so you can use any data-store you'd like, but Nubase provides helpers meant to be used with Kysely and Postgres, for example,
the Query Language (NQL). Nubase also has Postgres migration tools, even though you're not required to use it.
- There is no constraint as to how and where Nubase apps are deployed and served.

## How does Nubase work?

Nubase is provided as a set of packages. The main ones are `@nubase/backend` and `@nubase/frontend`. You create a Nubase app 
with the `npx @nubase/create my-app` command and everything is wired up automatically for you.

A Nubase app is a full-stack app, like a Next.js app. It contains a `src/common`, `src/backend` and a `src/frontend` directory.

We start by implementing the schema of the application in `src/common`. The schema is similar to Zod, but it has
features to help inform how forms, table, search are handled.

Once the schema is defined, we can now implement the backend endpoints. Nubase provides helpers to make sure the input
and output of your endpoints fit the schema, but the implementation of the backend is fully custom. You can implement 
whatever business logic you want. 


## Getting Started

The fastest way to create a Nubase app is to use the CLI:

```bash
npx @nubase/create my-app
cd my-app
npm run db:up      # Start PostgreSQL
npm run db:seed    # Seed with sample data
npm run dev        # Start development servers
```

This scaffolds a complete application with backend, frontend, and shared schema packages. See [Create a Nubase App](/docs/create-nubase-app) for more details.

## How does a Nubase app look like?

A Nubase app is basically a Vite app that renders a single top-level React component, the `NubaseApp` component.

It looks like:

```tsx
<NubaseApp config={config} />
```

## FAQ

### Can I integrate Nubase into an existing application?

No. With Nubase, your selected runtime (only React is available now) will run your application for you. The runtime will be responsible for routing, authentication and everything.
