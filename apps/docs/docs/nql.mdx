title: NQL — Nubase Query Language
description: Learn how to enable NQL on your endpoints and what users can type into a Nubase filter bar.

NQL is a small text query language that lets users express criteria like
`(Title CONTAINS "override" OR Title CONTAINS "vulnerability") AND Description STARTS_WITH "Bibo"`
directly in the filter bar. It complements the structured filter chips
covered in [Searching & Filtering](./searching.mdx): every search-capable
endpoint automatically accepts an `nql` parameter, and turning it on for
your endpoint is one declaration in your route handler.

This page is about **using** NQL in your app. For the design rationale
and pipeline internals, see [ADR 0008](./adrs/0008-nql-nubase-query-language.mdx).

## What users can type

NQL is case-insensitive, ASCII-only, and uses readable keyword operators
(no `~`, no wildcards baked into string values). The full grammar:

| Category | Operators |
| -------- | --------- |
| **Logical** | `AND`, `OR`, `NOT`, parentheses |
| **Numbers / dates** | `=`, `!=`, `<`, `<=`, `>`, `>=` |
| **Strings** | `=`, `!=`, `CONTAINS`, `STARTS_WITH`, `ENDS_WITH` |
| **Booleans** | `=`, `!=` |
| **Set membership** | `IN (v1, v2, …)`, `NOT IN (…)` |
| **Null** | `IS NULL`, `IS NOT NULL` (only on optional fields) |
| **Literals** | `"double-quoted strings"`, integers, decimals, signed numbers, `true`, `false`, `null` |

Precedence runs `NOT` &gt; `AND` &gt; `OR`, just like SQL. Use parentheses to
override.

A few notes that bite people coming from SQL:

- **All string comparisons are case-insensitive**, including `=` and `!=`.
  `title = "Foo"` matches a row with title `"foo"`. NQL compiles those
  to Postgres `ILIKE`, matching the JQL / KQL / Notion convention.
- **Field names are matched case-insensitively too** and canonicalised to
  the schema spelling. `Title`, `title`, and `TITLE` all resolve to the
  same field.
- **Quote your strings.** `title = foo` is a syntax error; `title = "foo"`
  is correct.
- **`= null` is rejected.** Use `IS NULL` / `IS NOT NULL`, and only on
  fields that are `nu.string().optional()` etc.
- **Empty `IN ()` lists are rejected.** Pass at least one literal.

### Examples

```nql
title CONTAINS "bug"
```

```nql
assigneeId IN (1, 2, 3) AND status != "closed"
```

```nql
description IS NULL OR description STARTS_WITH "WIP"
```

```nql
(title CONTAINS "override" OR title CONTAINS "vulnerability")
  AND description STARTS_WITH "Bibo"
```

## Enabling NQL on an endpoint

Three steps for any list endpoint that already uses
[`withSearchParams`](./searching.mdx) or `baseSearchParams`:

1. The `nql` request param is already there — `baseSearchParams` and
   `withSearchParams` both include it. You don't add anything to the
   schema.
2. In the handler, declare a **field bindings** map. This says "if NQL
   sees this schema field, use this Kysely column reference."
3. Call `compileNql(...)` on `params.nql` and apply the result to your
   query.

Below is the full integration from the questlog `getTickets` handler at
`apps/questlog/src/backend/api/routes/ticket.ts`:

```ts
import { compileNql, createNqlBindings, HttpError } from "@nubase/backend";
import { ticketListSchema } from "../../../common/resources/ticket";
import type { DB } from "../../db/db-types";   // generated by kysely-codegen
import { getDb } from "../../db/helpers/kysely";

// 1. Declare the queryable universe once, at module load.
//    - Keys must exist in `ticketListSchema._shape`.
//    - Values must be valid Kysely column refs on `DB`.
//    - Cross-table refs (here `users.*`) are only valid if the query
//      below joins those tables.
const ticketListNqlBindings = createNqlBindings<DB>()(ticketListSchema, {
  id:            "tickets.id",
  title:         "tickets.title",
  description:   "tickets.description",
  assigneeId:    "tickets.assigneeId",
  assigneeName:  "users.displayName",
  assigneeEmail: "users.email",
});

export const ticketHandlers = {
  getTickets: createHandler((e) => e.getTickets, {
    auth: "required",
    handler: async ({ params, ctx }) => {
      const workspace = ctx.get("workspace");
      const db = getDb();

      // 2. Build your query as usual — joins, workspace isolation, etc.
      let query = db
        .selectFrom("tickets")
        .leftJoin("users", "tickets.assigneeId", "users.id")
        .select([/* ...columns... */])
        .where("tickets.workspaceId", "=", workspace.id);

      // 3. If NQL is present, compile and apply it. Errors come back
      //    structured so the frontend can render them inline.
      if (params.nql && params.nql.trim() !== "") {
        const compiled = compileNql(params.nql, ticketListSchema, {
          fields: ticketListNqlBindings,
        });
        if (!compiled.ok) {
          throw new HttpError(400, compiled.error.message, {
            code:   compiled.error.code,
            line:   compiled.error.line,
            column: compiled.error.column,
            length: compiled.error.length,
          });
        }
        query = query.where(compiled.value);
      }

      // ...other structured filters (q, title, assigneeId, ...) still work.

      return await query.execute();
    },
  }),
};
```

That's the whole integration. From this point on, any client that sends
`?nql=…` to this endpoint gets the parsed expression compiled into the
WHERE clause, with full validation against `ticketListSchema` and the
bindings map.

### Why bindings are required

If NQL just took schema field names and converted them to column names by
convention, joined endpoints would silently break. The bindings map is
the **single declaration of how a schema field reaches SQL**. It also
provides three layers of protection:

1. **At compile time** — `createNqlBindings<DB>()` rejects keys that
   aren't in the schema (`Object.keys` of `_shape`) and values that
   aren't valid `StringReference<DB, TB>` for your generated Kysely DB
   type. Typo `"users.disaplyName"` and TypeScript fails the build.
2. **At handler-module load** — a runtime check throws if a key drifts
   out of the schema (e.g. someone `.omit()`ed a field but forgot the
   bindings).
3. **At request time** — only fields whose keys appear in `bindings` are
   queryable. Asking for a field that isn't bound returns a `VALIDATE`
   error to the user; columns the handler chose to keep private stay
   private.

### Choosing the schema you bind against

Use the same schema your endpoint exposes as `requestParams`. For list
views with joins (where the response includes columns from joined
tables), that's the *list* schema, not the base one — that way the
queryable universe matches what the user sees in the table. In the
questlog example, the request and the bindings both use
`ticketListSchema`, which has `assigneeName` / `assigneeEmail` in
addition to the base ticket fields.

## How errors reach the user

When NQL compilation fails the handler throws an `HttpError(400, …)`
whose JSON body looks like:

```json
{
  "error":   "Expected comparison operator after 'testing', got end of input",
  "code":    "PARSE",
  "line":    1,
  "column":  8,
  "length":  null
}
```

`code` is one of:

- `TOKENIZE` — the user typed an unknown character, an unterminated
  string, or an invalid escape.
- `PARSE` — the syntax is wrong: missing operator, missing closing
  paren, trailing `AND` with no right-hand side, …
- `VALIDATE` — the syntax is fine but the query refers to a field that
  doesn't exist or isn't bound, mixes a string operator with a number
  field, uses `IS NULL` on a required field, etc.

`line` and `column` are 1-based and point at the offending token. The
filter bar uses these to render the error message inline under the NQL
editor; you don't need to handle this in your view config — it's wired
in `ResourceSearchViewRenderer`.

## What happens on the frontend

Users get the NQL UI for free on any view whose `schemaFilter` includes
the `nql` field — i.e. any view backed by `withSearchParams`. The filter
bar shows a `Filters | NQL` toggle:

- **Filters mode** (default) — the structured chips described in
  [Searching & Filtering](./searching.mdx).
- **NQL mode** — the chips are replaced with a Monaco-backed text editor
  with syntax highlighting and schema-driven autocomplete for field
  names, keywords, and operator snippets. The value is debounced
  (~400 ms) before it's sent as `?nql=…`, and the inline error message
  from a 400 response renders directly under the editor.

Switching between modes is mutually exclusive on the wire — Filters mode
sends `q` and structured fields; NQL mode sends only `nql`. While the
user is editing an invalid NQL query, the previous successful row set
stays visible in the data grid.

## Limitations

NQL deliberately stops at a small grammar. Things that are **out of
scope** today:

- **Relationship traversal.** You can't write `assignee.team.name CONTAINS "x"`. If you need that, expose the joined column as a flat
  field on your list schema and bind it.
- **Stored / named queries.** NQL describes a single criterion; saving
  it as "My critical bugs" is a UI concern, not a language feature.
- **Operators we haven't added yet.** No `BETWEEN`, no `LIKE` with
  explicit `%` / `_` wildcards, no regex, no date-math literals
  (`-1w`).
- **Client-side syntax errors before submission.** The editor highlights
  syntax but doesn't pre-parse — invalid queries are caught after the
  request returns 400. (This is a planned follow-up: tokenize / parse /
  validate are pure and will move into `@nubase/core` so the editor can
  pre-flight them.)

For everything else — disjunctions, grouping, IN, NOT, IS NULL, mixed
field types — NQL is the intended tool.
