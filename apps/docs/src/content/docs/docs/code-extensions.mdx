The Code Extensions System enables developers to extend Nubase with custom logic — hooks, validations, endpoints, and actions — without modifying the platform itself. The database is the source of truth for schema, and code extends behavior only.

## Architecture Overview

Nubase is a low-code platform where:
- The **DB is the source of truth** for schema (tables, columns, types)
- **Code extends behavior** via hooks, validations, custom endpoints, and actions
- Developers work in a local TypeScript project with full type-safety
- A CLI `pull`/`push` workflow syncs with the Nubase server
- Custom code runs in **V8 isolates** on the server

```
┌──────────────────────────────┐
│   Developer's Local Project  │
│                              │
│  nubase.config.ts            │
│  src/index.ts (defineApp)    │
│  .nubase/types/ (generated)  │
└──────────┬───────────────────┘
           │ nubase push
           ▼
┌──────────────────────────────┐
│      Nubase Server           │
│                              │
│  ┌─────────────────────────┐ │
│  │  V8 Isolate Runtime     │ │
│  │  (hooks, endpoints,     │ │
│  │   actions, validations) │ │
│  └──────────┬──────────────┘ │
│             │ bridge calls   │
│  ┌──────────▼──────────────┐ │
│  │  Generic CRUD Handlers  │ │
│  │  (auto-generated)       │ │
│  └──────────┬──────────────┘ │
│             │                │
│  ┌──────────▼──────────────┐ │
│  │  Data DB + Nubase DB    │ │
│  └─────────────────────────┘ │
└──────────────────────────────┘
```

| Decision | Choice |
|----------|--------|
| Source of truth for schema | Database (Nubase DB metadata + Data DB structure) |
| Code's role | Extend behavior only (hooks, validations, endpoints, actions) |
| App structure | Config-based with `defineApp()` and full type-safety |
| Type generation | `nubase pull` generates `.nubase/types/` (like Prisma) |
| Code runtime | V8 isolates (`isolated-vm`) |

## @nubase/sdk Package

Published as `@nubase/sdk`. The single dependency for app developers.

### `defineApp()` — The Entry Point

```typescript
import { defineApp } from "@nubase/sdk";
import type { NubaseEntities } from "../.nubase/types"; // Generated by `nubase pull`

export default defineApp<NubaseEntities>({
  hooks: { ... },
  validations: { ... },
  endpoints: { ... },
  actions: { ... },
  computedFields: { ... },
});
```

The generic parameter `TEntities` flows through everything — hook names are typed as `${entityName}:${hookType}`, contexts know the entity type, validations receive typed data.

### Hooks

8 hook types covering the full CRUD lifecycle:

```typescript
type HookType = "before-create" | "after-create" | "before-update" | "after-update"
              | "before-delete" | "after-delete" | "before-read" | "after-read";
```

Hook keys are fully typed from entity names: `"ticket:before-create"`, `"ticket:after-update"`, etc.

Context varies by hook type:

| Hook Type | Context Fields |
|-----------|---------------|
| `before-create` | `data` (the row about to be inserted) |
| `after-create` | `data`, `result` (the created row) |
| `before-update` | `id`, `changes` (partial), `existing` (current row) |
| `after-update` | `id`, `changes`, `existing`, `result` |
| `before-delete` | `id`, `existing` |
| `after-delete` | `id`, `existing` |
| `before-read` | `id?`, `filters?` |
| `after-read` | `id?`, `result` (row or rows) |

Every context includes: `user`, `workspace`, `db` (typed DatabaseClient), `log`.

Before-hooks can return `{ abort: { statusCode, message } }` to prevent the operation, or `{ data: {...} }` to modify the data.

```typescript
hooks: {
  "ticket:before-create": async (ctx) => {
    if (!ctx.data.assignee_email) {
      return { data: { ...ctx.data, assignee_email: ctx.user.email } };
    }
  },
  "ticket:after-create": async (ctx) => {
    ctx.log(`Ticket #${ctx.result.id} created`);
  },
}
```

### Validations

Per-field and entity-level validators with full type safety:

```typescript
validations: {
  ticket: {
    fields: {
      title: (value) => value.length < 3 ? "Too short" : undefined,
    },
    entity: (data, ctx) => {
      if (ctx.isCreate && !data.title) {
        return [{ field: "title", message: "Required" }];
      }
    },
  },
}
```

### Custom Endpoints

Mounted at `/api/custom/<path>` on the server:

```typescript
endpoints: {
  "ticket-summary": {
    method: "GET",
    path: "/ticket-summary",
    handler: async (ctx) => {
      const total = await ctx.db.count("ticket");
      return { total };
    },
  },
}
```

### Custom Actions

UI-triggered operations with single, bulk, or global scope:

```typescript
actions: {
  "archive-ticket": {
    entity: "ticket",
    label: "Archive",
    scope: "bulk",
    handler: async (ctx) => {
      for (const id of ctx.selectedIds) {
        await ctx.db.delete("ticket", id);
      }
      return { success: true, message: "Archived", refreshEntities: ["ticket"] };
    },
  },
}
```

### DatabaseClient

A type-safe query API available in all contexts. Each call is serialized across the V8 isolate bridge — NOT raw SQL.

```typescript
interface DatabaseClient<TEntities> {
  findById(entity, id): Promise<Row | null>;
  findMany(entity, options?): Promise<Row[]>;
  count(entity, where?): Promise<number>;
  create(entity, data): Promise<Row>;
  update(entity, id, data): Promise<Row>;
  delete(entity, id): Promise<boolean>;
}
```

The `where` clause supports operators: `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `in`, `like`, `isNull`.

## Type Generation

### Server Endpoint

`GET /api/nubase/schema` returns schema metadata for the Data DB, derived from `@nubase/pg`'s `extractSchema()`. The response includes a `schemaVersion` hash for drift detection.

### Generated Files

`nubase pull` fetches the schema and generates `.nubase/types/`:

```
.nubase/
  types/
    index.ts           # Barrel export
    entities.ts        # All entity type re-exports
    entity-map.ts      # NubaseEntities type (the generic param for defineApp)
    ticket.ts          # interface Ticket { id: number; title: string; ... }
  schema-metadata.json # Cached raw metadata
```

**PG to TypeScript type mapping:**

| PostgreSQL | TypeScript |
|-----------|-----------|
| `integer`, `serial`, `bigint`, `float`, `numeric` | `number` |
| `varchar`, `text`, `uuid` | `string` |
| `boolean` | `boolean` |
| `timestamp`, `date`, `time` | `string` (ISO) |
| `jsonb`, `json` | `unknown` |
| User-defined enum | Union of string literals |

Each entity generates three interfaces: `Ticket` (row), `TicketInsert` (omit id, optionals for defaults), `TicketUpdate` (all partial).

## CLI Workflow

### `nubase pull`

1. Read `nubase.config.ts` for server URL + auth token
2. `GET /api/nubase/schema`
3. Generate `.nubase/types/` files
4. Cache `schema-metadata.json`

### `nubase push`

1. Read `nubase.config.ts`
2. Bundle with **esbuild** (target: es2022, format: esm, external: `@nubase/sdk`)
3. Extract manifest from bundle (hook names, endpoints, actions)
4. Validate: entity names exist, hook keys valid, no endpoint path conflicts
5. Validate: `schemaVersion` matches server (reject if stale — "run `nubase pull` first")
6. Upload: `POST /api/nubase/apps/deploy` with `{ bundle, sourceMap, manifest, checksum }`

### `nubase.config.ts`

```typescript
import { defineConfig } from "@nubase/sdk/config";

export default defineConfig({
  server: {
    url: process.env.NUBASE_SERVER_URL ?? "http://localhost:3001",
    token: process.env.NUBASE_API_TOKEN,
  },
  workspace: "tavern",
  output: { typesDir: ".nubase/types" },
  app: { entry: "src/index.ts" },
});
```

## Server-Side Architecture

### API Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/nubase/schema` | Schema metadata for type generation |
| `POST` | `/api/nubase/apps/deploy` | Upload and activate app bundle |
| `GET` | `/api/nubase/apps/deployments` | Deployment history |
| `POST` | `/api/nubase/apps/deployments/:id/activate` | Rollback to previous deployment |
| `GET` | `/api/nubase/apps/status` | Current deployment status |

### V8 Isolate Runtime

Custom code runs in `isolated-vm` V8 isolates:

```
[Hono Request] → [Hook Trigger] → [Isolate Pool] → [V8 Isolate]
                                                         ↕
                                                   [Bridge Functions]
                                                   (db queries, logging)
```

- **Isolate Pool**: Pre-warmed isolates per workspace, recycled after N executions
- **Bridge**: Each `db.findById()` call serializes to a message, the host executes the Drizzle query, returns the result
- **Limits**: 128MB memory, 5s CPU per execution
- **Sandboxing**: No `fs`, `process`, `require`, `eval`. Only: `JSON`, `Date`, `Math`, `crypto.randomUUID`, `console.log` (→ structured log), optionally `fetch` with domain allowlist

### Generic CRUD Handlers

Auto-generated CRUD for all Data DB entities with hook integration:

```
POST   /api/:entity     → validate → before-create hook → INSERT → after-create hook
GET    /api/:entity      → before-read hook → SELECT → after-read hook → compute fields
GET    /api/:entity/:id  → before-read hook → SELECT → after-read hook → compute fields
PATCH  /api/:entity/:id  → validate → before-update hook → UPDATE → after-update hook
DELETE /api/:entity/:id  → before-delete hook → DELETE → after-delete hook
```

## Database Schema

New tables in `nubase_db` for the extensions system:

### `app_deployments`

Stores uploaded JS bundles per workspace.

| Column | Type | Description |
|--------|------|-------------|
| `id` | `serial` | Primary key |
| `workspace_id` | `integer` | FK to workspaces |
| `version` | `integer` | Auto-incrementing per workspace |
| `schema_version` | `varchar(64)` | SHA-256 hash of schema at deploy time |
| `bundle` | `text` | The compiled JS bundle |
| `source_map` | `text` | Source map for debugging |
| `manifest` | `jsonb` | Extracted hooks, endpoints, actions |
| `checksum` | `varchar(64)` | Bundle checksum |
| `is_active` | `boolean` | Whether this is the current deployment |
| `deployed_by` | `integer` | User ID who deployed |
| `deployed_at` | `timestamptz` | Deployment timestamp |

### `hook_registrations`

Denormalized from manifest for fast CRUD lookup.

| Column | Type | Description |
|--------|------|-------------|
| `id` | `serial` | Primary key |
| `workspace_id` | `integer` | FK to workspaces |
| `deployment_id` | `integer` | FK to app_deployments |
| `hook_key` | `varchar(255)` | Full key like `ticket:before-create` |
| `entity_name` | `varchar(255)` | Entity name extracted from key |
| `hook_type` | `varchar(50)` | Hook type extracted from key |

### `custom_endpoint_registrations`

Denormalized from manifest for route registration.

| Column | Type | Description |
|--------|------|-------------|
| `id` | `serial` | Primary key |
| `workspace_id` | `integer` | FK to workspaces |
| `deployment_id` | `integer` | FK to app_deployments |
| `method` | `varchar(10)` | HTTP method |
| `path` | `varchar(255)` | Endpoint path |

### `entity_metadata`

UI metadata for entities (display names, icons, field labels, layouts).

| Column | Type | Description |
|--------|------|-------------|
| `id` | `serial` | Primary key |
| `workspace_id` | `integer` | FK to workspaces |
| `table_name` | `varchar(255)` | The DB table name |
| `display_name` | `varchar(255)` | Human-readable name |
| `description` | `text` | Entity description |
| `icon` | `varchar(100)` | Icon identifier |
| `metadata` | `jsonb` | Additional UI metadata |

All tables have RLS following the existing `app.current_workspace_id` pattern.

## Migration Path

### What moves out of the platform

| Current Location | Destination |
|-----------------|-------------|
| `apps/nubase/schema/src/resources/ticket.ts` | DB schema + entity_metadata |
| `apps/nubase/schema/src/endpoints/ticket/` | Auto-generated generic CRUD |
| `apps/nubase/backend/src/api/routes/ticket.ts` | Generic CRUD + hooks from SDK |
| `apps/nubase/backend/src/db/schema/ticket.ts` | Dynamic schema introspection |

### What stays

- `createResource()` stays in `@nubase/frontend` as internal rendering engine
- `SchemaResourceFactory` generates `ResourceDescriptor` from entity metadata
- `packages/core` schema system stays for form rendering
- `packages/pg` stays for schema extraction
- User resource stays hand-written (users are in nubase_db, not data_db)

### Transition strategy

Hand-written resources take precedence over auto-generated. Gradual migration.

## Edge Cases

1. **Schema drift**: `schemaVersion` hash check on push prevents deploying stale code
2. **Hook loops**: Track call depth, abort at configurable limit (e.g., 10)
3. **Hook failures**: Before-hooks abort operation; after-hooks log error but don't affect response
4. **Bundle size**: 5MB limit enforced at upload
5. **Isolate warm-up**: Pool pre-warms on deployment activation
6. **Cross-DB references**: Type generator recognizes FK-like email columns, annotates as lookup
7. **User entity**: Lives in nubase_db, not auto-generated. Stays hand-written
8. **Dev mode**: `nubase dev` watches files, auto-bundles, pushes to dev endpoint for fast iteration
